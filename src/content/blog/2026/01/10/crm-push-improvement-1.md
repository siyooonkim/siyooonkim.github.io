마케팅의 핵심 중 하나는 CRM이다.<br>
CRM (Customer Relationship Management)은 유저를 유입시키는 것에서 끝나는 게 아니라, 이후의 행동을 설계하고 이어가는 과정에 가깝다.<br>

유입된 유저를 대상으로 팝업, 쿠폰, 푸시 알림 같은 다양한 수단을 활용해 구매 → 재구매 → 충성 고객으로 이어지는 흐름을 만든다.<br>

---

## 문제 정의 

현재 서비스에서도 CRM을 활용해 유저 행동에 기반한 알림을 보내고 있었다.<br>

예를 들어, '장바구니에 담기'나 '좋아요'와 같은 액션을 한 상품의 가격이 내려가면 푸시 알림을 발송하는 방식이다.<br>

한가지 문제는, 이 구조가 생각보다 단순했다는 점이다.<br>

조건이 충족되면 알림을 보내는 수준이었고,<br>
유저 상태나 상품 변화에 따라 유연하게 대응하지 못했다.<br>

상품 할인 이벤트가 발생하면, 해당 상품을 장바구니에 담고 있는 유저 전원에게 즉시 푸시를 발송하는 구조였다.<br>
상품 단위로 발송되다 보니, **유저 관점에서의 제어가 부족**했다.<br>

예를 들어, 한 유저가 장바구니에 상품 5개를 담고 있는 상황에서 이 상품들이 **동시에 할인**되면,<br>
여러 이벤트 중 하나가 무작위로 선택되어 푸시가 발송되는 식이었다.<br>

이커머스 회사를 다녀본 사람은 당연히 알겠지만 연말, 연초, 블랙프라이데이, 클리어런스 등등 정기적으로 상품 할인을 진행한다.<br>

이 때, 브랜드에서 꼴랑 하나의 상품만 하는 경우는 드물고, 여러 상품을 동시에 진행하기 마련이다.<br>

즉, 여러 상품들이 동시에 할인되는 경우는 굉장히 흔한 경우이고,<br>
이 상황에서 푸시알림에 동의한 유저에게 더 큰 임팩트를 줄 수 있다면 이를 활용하는게 방법이다.<br>

결국 필요한 것은 상품이 아닌, **유저 단위의 제어**다.<br>

여러 상품이 동시에 할인되더라도, 한 유저에게는 가장 효과가 높은 알림 1건만 우선적으로 보내고,<br>
이후에는 일정 간격을 두고 순차적으로 발송하는 방식이 더 적절하다.<br>

예를 들어, 유저가 A, B, C, D, E 상품을 담아둔 상태에서 모두 할인되었다고 가정하자.<br>
각 상품에 대해 할인율과 최근 주문량 등을 기반으로 점수를 매긴다.<br>

```
A: 50점
B: 40점
C: 30점
D: 20점
E: 10점
```

가장 점수가 높은 A를 먼저 발송하고, 이후 일정 간격을 두고(ex.3일) B, C 순으로 순차 발송한다.<br>

너무 많은 알림을 보내게되면 오히려 유저입장에서 알림은 정보가 아닌 노이즈가 되고 효과가 떨어지고만다.<br> 

---

## 우선순위 설계

"가장 효과 좋은 1건"이 뭔지 정의해야 했다.<br>

### gap = 현재 할인율 - 원래 할인율

원래 할인율이 40%었는데, 블랙 프라이데이 기념으로 55%까지 할인율을 낮췄다고 가정해보자.<br>

그럼 gap은 15다.<br>

gap이 클수록 유저 입장에서 "이거 싸졌다"는 느낌이 강하다.<br>

중요한 건 **이벤트 시점이 아니라 발송 시점에 다시 계산**한다는 점이다.<br>

```
Day 0: 이벤트 발생. A 상품 40% → 50% (gap=10)
Day 2: A 상품 할인율이 45%로 줄어듦

발송 시점 gap = 45 - 40 = 5 (gap =5)
```

### 확정한 규칙

| 항목 | 결정 |
|------|------|
| 우선순위 | gap 큰 순, 동률이면 3개월 판매량 순 |
| 발송 주기 | 유저당 3일에 1건 |
| 발송 시점 | 발송 시점 기준 재계산 |
| 발송 조건 | current > before |
| 원복 | current <= before면 발송 안 함 |

---

## 할인율을 3개 저장하는 이유

발송 판단에 필요한 할인율이 3개다.

- **before**: 이벤트 발생 전 할인율
- **after**: 이벤트 발생 시점 할인율
- **current**: 발송 시점 실제 할인율

여기에서 **`current <= before`면 할인 기간이 끝난거기 때문에 알림 대상에서 제외**해야한다.<br>

반대로 같은 상품이 나중에 더 할인되는 경우(새 이벤트의 before > 기존 sent의 after), 재발송을 허용한다.<br> 

---

## 구조 설계 

결국 우리에게 필요한건 정해진 기간마다 할인율 gap을 재계산하여 rank의 1순위를 전달하는 것이다.<br>

기술적으로 중요한건 정해진 기간까지 후보 정보를 저장해놨다가, 재계산 시점이 오면 다시 gap을 계산하는 구조를 만들어야 한다.<br>

Typescript 기반의 Bull Queue + Redis 구조를 활용했다.<br>

## 분산 락

### 낙관적 락 vs 비관적 락

낙관적 락(Optimistic Lock)은 **충돌이 자주 발생하지 않는다는 가정** 하에, 일단 작업을 수행한 뒤 **커밋 시점에 데이터 변경 여부를 검증하는 방식**이다. 주로 version 컬럼이나 timestamp를 이용해 “내가 읽은 이후에 다른 변경이 있었는지”를 확인한다.

반면 비관적 락(Pessimistic Lock)은 충돌 가능성을 전제로, **작업을 시작하기 전에 대상 리소스를 선점(lock)**하고,
**처리가 끝날 때까지 다른 트랜잭션의 접근을 제한하는 방식**이다.

이 문제에서 중요한 포인트는 푸시 발송이 비가역적인 작업이라는 점이다.<br>

푸시알림은 한 번 외부 시스템으로 전송되면 취소하거나 롤백할 수 없다.<br>

따라서 낙관적 락을 사용해 "일단 처리하고 나중에 충돌을 감지"하는 방식은 적합하지 않다.<br>

충돌을 감지하는 시점에는 이미 푸시가 발송된 이후이기 때문에, DB 레벨에서의 롤백이나 unique constraint로는 문제를 해결할 수 없다.<br>

그래서 유저 단위로 락을 선점해 동시 실행을 막는, 즉 비관적 락 기반의 직렬 처리 방식을 선택했다.<br>


### 락 경합과 인프라 장애 구분

초기 구현에서는 락 획득 과정에서 발생하는 모든 예외를 동일하게 처리하고 있었다.<br> 

```ts
try {
  const lock = await this.redis.lock(key);
} catch (error) {
  return null; // 모든 예외를 락 실패로 처리
}
```

문제는 이 방식이 정상적인 락 경합과 비정상적인 인프라 장애를 구분하지 못한다는 점이다.<br>

예를 들어, 다른 워커가 이미 락을 선점한 경우라면 단순히 이번 작업을 건너뛰면 된다.<br>
하지만 Redis 연결 실패나 타임아웃까지 동일하게 처리해버리면, 실제로는 아무도 락을 잡지 못하고 있는데도 시스템은 이를 "이미 다른 워커가 처리 중"인 정상 상황으로 오인하게 된다.<br>

그 결과 장애가 발생해도 감지되지 않고, 특정 작업이 조용히 유실될 수 있다.<br>

따라서 아래와 같이 락 경합은 정상적인 제어 흐름으로 처리하고,<br>
그 외 예외는 인프라 장애로 간주해 별도로 대응하도록 분리했다.<br>

- 락 경합: 다른 워커가 이미 락을 보유 중인 상태. 정상 상황이므로 스킵 가능
- 인프라 장애: Redis 연결 실패, 네트워크 타임아웃 등. 비정상 상황이므로 로깅·알림 필요


```ts
private isLockContentionError(error: unknown): boolean {
  return (
    error instanceof ResourceLockedError ||
    (error instanceof Error && error.name === 'ResourceLockedError')
  );
}
```

이처럼 분리해두면, 락을 얻지 못한 이유가 단순 경합인지 아니면 시스템 장애인지 명확하게 판단할 수 있고,<br>
운영 중 장애 탐지와 복구 대응도 쉬워진다.<br>

**(푸시 알림을 직접 받지 못하는 상황에서는 이와같은 예외처리 중요도가 더 올라간다.)**

### 락 실패 시 candidate 방치 방어 

락을 획득하지 못했다고 해서, 해당 유저의 candidate가 그대로 방치되면 안 된다.<br>

예를 들어 Worker A가 락을 선점해 발송을 처리하는 동안, Worker B가 같은 유저의 다른 candidate를 저장했지만 락 획득에는 실패할 수 있다.<br>

이 경우 candidate 자체는 Redis에 남아 있어도, 이후 이를 다시 처리할 job이 없다면 영원히 발송 대상에서 누락될 수 있다.<br>

이를 막기 위해 락 획득에 실패한 유저에 대해서도 pending job 존재 여부를 확인하고,<br>
아직 예약된 작업이 없다면 다음 delayed job을 등록하도록 했다.<br>

```
await Promise.allSettled(
  lockFailedUserIds.map(async (userId) => {
    const hasPendingJob = await this.nextSendJobService.hasPendingJob(userId);
    if (!hasPendingJob) {
      await this.nextSendJobService.scheduleNext(userId);
    }
  }),
);
```

### 2차 방어

분산 락만으로 대부분의 중복 발송은 막을 수 있지만,<br>
분산 시스템 구조에서는 edge case가 종종 발생한다.<br>

예를 들어 락 TTL이 만료되기 직전에 worker가 느려지면,<br>
첫 번째 worker가 아직 작업을 끝내지 못한 상태에서 두 번째 worker가 같은 유저의 락을 다시 획득할 가능성이 있다.<br>

이 경우 두 worker가 동시에 발송을 시도할 수 있다.<br>

이런 상황까지 100% 차단하는 것은 현실적으로 어렵다.<br>
그래서 이 구간에서는 완벽한 차단보다 빠른 감지와 운영 대응에 초점을 맞췄다.<br>

발송 기록을 저장할 때 이전 기록과 비교하여 이미 더 크거나 같은 할인율 기준으로 발송된 이력이 있다면 중복 가능성으로 판단해 알림을 보내도록 설정했다.<br>

## 유저 단위 순차 발송 흐름

유저가 장바구니에 A, B, C 상품을 담고 있고, 세 상품이 동시에 할인되었다고 가정하자.<br>

```
Day 0:
  A: before=40%, after=50% → gap=10
  B: before=20%, after=30% → gap=10
  C: before=10%, after=35% → gap=25

  → 세 상품 전부 candidate에 저장
  → current 조회 → gap 재계산 → C(gap=25)가 1위
  → C 발송 → 3일 뒤 delayed job 예약

Day 3 (delayed job 실행):
  → candidate 재조회: A, B 남아있음
  → current 재조회
  → A(current=50%, gap=10)
  → B(current=28%, gap=8)

  → A가 1위이므로 발송
  → sent record 저장
  → 남은 후보 B에 대해 다시 3일 뒤 delayed job 예약

Day 6 (delayed job 실행):
  → candidate 재조회: B만 남아있음
  → current 재조회: B(current=25%, gap=5)

  → B 발송
  → sent record 저장
  → 남은 후보가 없으므로 종료
```

Day 0에 계산한 gap이 Day 3에도 유효하리란 보장이 없기 때문에,<br>
매번 발송 시점에 current를 다시 조회한다.<br>

## 예외 처리

위처럼 하면 대부분의 예외가 발송 시점 재평가 과정에서 자연스럽게 걸러진다.<br>

### 할인 원복

```
Day 0: A(before=40%, after=50%) → candidate 저장
Day 2: A 할인 종료, 가격 원복 → current=40%
Day 3: current(40%) <= before(40%) → 발송 안 함 → 다음 후보로
```

### 장바구니 삭제 / 구매

```
Day 0: B 발송 대기 중
Day 1: 유저가 B를 장바구니에서 삭제 (또는 구매)
Day 3: 장바구니 재조회 → B 없음 → 후보 제외
```

### 대기 중 더 큰 할인 후보 등장

```
Day 0: A(gap=10) 발송, 3일 뒤 예약
Day 2: D(before=20%, after=50%) 새 이벤트 발생
       → candidate 저장
       → pending job이 이미 있음 → 즉시 발송 스킵

Day 3: D(gap=28) > B(gap=10) → D 발송
```

Day 2에 새 이벤트가 와도 즉시 보내지 않는다.<br>
Day 3에 전체 후보를 재계산하면 D가 자연스럽게 선택된다.<br>

## 마무리
이번 작업을 진행하며, 조건을 조합해 푸시를 제어하는 방식이 과연 최선인지,<br>
또 기술적으로도 Queue + Redis 구조가 가장 적절한 선택이었는지에 대해서는 아직 100% 확신을 가지고 있지는 않다.<br>

Mongo 기반으로 더 단순하게 풀 수도 있었을 것 같고, 애초에 다른 방향의 운영 전략이 더 나았을 가능성도 충분히 있다.<br>

(더 나은 방법이 보인다면 언제든지 개선할 여지는 열려 있다.)<br>

다만 이번 작업에서 중요했던 건 **기존에 관성적으로 반복되던 액션에 의문을 던지고**,<br>
디테일을 개선해 **실제 유저 경험과 전환율을 끌어올리는 과정**이었다.<br>

단순히 기능을 구현하는 데서 끝나는 것이 아니라,<br>
서비스의 흐름을 이해하고 "왜 이 방식이어야 하는지"를 계속 고민하는 것.<br>

**앞으로도 그런 방향으로 진짜 문제를 인지하고 더 나은 솔루션을 제시하는 개발자가 되고 싶다.**<br>

마지막으로, 구조를 설계하는 과정에서 Bull Queue, RabbitMQ, Kafka 관련 영상을 보게 되었는데,<br>
몰랐던 개념들을 많이 알게 되었고 꽤 흥미로웠다.<br>
