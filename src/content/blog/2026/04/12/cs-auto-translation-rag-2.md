[이전 편](/blog/2026/04/13/cs-auto-translation-rag-1)에서 요구사항을 정리하고 유저 플로우를 설계했다. 이번 편에서는 AI 부분을 설계한다.

핵심 질문은 하나였다. **AI가 답변을 추천할 때, 브랜드 정책과 응대 지식을 어떻게 주입할 것인가?**

---

## 두 가지 선택지

### 선택지 A: RAG (Retrieval-Augmented Generation)

RAG는 요즘 LLM 활용에서 자주 등장하는 패턴이다. 간단히 설명하면 이렇다.

```
1. 사내 문서(FAQ, 정책 가이드, 과거 응대 로그)를 벡터 DB에 저장
2. 고객 질문이 들어오면 → 벡터 검색으로 관련 문서를 찾음
3. 찾은 문서 + 질문을 LLM에 함께 전달
4. LLM이 문서 내용을 참고해서 답변 생성
```

장점은 분명하다. 문서가 많아져도 관련 있는 것만 골라서 프롬프트에 넣으니, 토큰 낭비가 적고 확장성이 좋다. 수백~수천 개의 정책 문서가 있는 대규모 CS센터라면 RAG가 맞다.

### 선택지 B: DB 정책 테이블 직접 주입

RAG를 쓰지 않고, 카테고리별 정책을 DB 테이블에 저장해두고 쿼리로 꺼내서 프롬프트에 직접 넣는 방식이다.

```
1. CsPolicy 테이블에 카테고리별 정책 저장 (배송 기간, 반품 조건 등)
2. 고객 문의가 들어오면 → 해당 카테고리의 정책을 DB에서 조회
3. 공통 정책 + 카테고리 정책을 프롬프트에 직접 주입
4. LLM이 정책을 참고해서 답변 생성
```

---

## 왜 RAG를 안 썼나

**우리 상황에서 RAG는 오버스펙이었다.**

### 정책 문서가 많지 않다

RAG가 빛나는 건 참조해야 할 문서가 수백~수천 건일 때다. 벡터 검색으로 관련 문서를 골라내는 게 의미가 있으려면 그만큼 문서 풀이 커야 한다.

우리 정책은 카테고리 7개, 카테고리당 2~3개. 전체 **20개 남짓**이다. 벡터 검색 없이 카테고리 기준 DB 쿼리 한 번이면 필요한 정책을 전부 꺼낼 수 있다.

### 인프라 오버헤드

RAG를 쓰려면 필요한 것들:

| RAG에 필요한 것 | 우리 상황 |
|----------------|----------|
| 벡터 DB (Pinecone, Weaviate 등) | 새로운 인프라 추가 |
| 임베딩 파이프라인 | 문서 변경 시 재임베딩 |
| 청킹 전략 설계 | 문서를 어떻게 자를지 |
| 검색 품질 튜닝 | top-k, 유사도 임계값 |

20개 정책을 위해 이 인프라를 전부 세우는 건 과하다.

### 정확도 문제

RAG는 "관련 있는 문서를 검색"하는 단계가 있다. 이 검색이 100% 정확하지 않다. 벡터 유사도 기반이라, 의미적으로 비슷하지만 실제로는 다른 정책을 가져올 수 있다.

우리 경우는 **고객이 카테고리를 직접 선택**한다. 배송을 선택하면 배송 정책을, 반품을 선택하면 반품 정책을 주면 된다. 검색할 필요가 없다. 카테고리가 곧 검색 키다.

### 비교 정리

| 기준 | RAG | DB 정책 주입 |
|------|-----|-------------|
| 정책 규모 | 수백~수천 건 | **수십 건** |
| 인프라 | 벡터 DB + 임베딩 파이프라인 | **기존 MySQL** |
| 검색 정확도 | 벡터 유사도 (오검색 가능) | **카테고리 = 검색 키** |
| 정책 업데이트 | 재임베딩 필요 | **DB UPDATE 한 줄** |
| 구현 복잡도 | 높음 | **낮음** |
| 토큰 사용량 | 적음 (관련 문서만) | 약간 많음 (카테고리 전체) |

토큰은 RAG가 유리하지만, 카테고리당 정책이 2~3개에 각 1~2문장이라 전부 넣어도 ~100토큰이다. 절약할 의미가 없다.

## 선택한 구조: CsPolicy 테이블

정책을 DB 테이블에 넣고, 카테고리별로 조회해서 프롬프트에 주입한다.

```sql
-- 공통 정책 (모든 카테고리에서 참조)
INSERT INTO CsPolicy (categoryId, title, contentKo) VALUES
(null, '영업시간', '평일 10:00~13:00, 14:30~17:00. 토일공휴일 휴무'),
(null, '결제수단', '크레딧카드, 편의점결제, PayPay 지원');

-- 배송 정책 (categoryId=3)
INSERT INTO CsPolicy (categoryId, title, contentKo) VALUES
(3, '배송 기간', '주문 확정 후 2-3일 내 발송. 국제배송 10-12일. 총 약 14일'),
(3, '배송 추적', '발송 완료 후 추적번호 메일 안내'),
(3, '배송 지연', '천후, 통관에 따라 지연 가능. 지연 시 별도 안내');
```

고객이 "배송"을 선택하면:

```
공통 정책 2개 + 배송 정책 3개 = 5개만 프롬프트에 주입 (~100토큰)
```

정책을 바꾸고 싶으면 DB에서 UPDATE 한 줄이면 된다. 코드 배포 필요 없다.

## AI 답변 추천 동작

### 프롬프트 구성

```
[시스템 프롬프트 - 고정]
당신은 amondz의 일본 고객 CS 전문 상담 AI입니다.
일본어 경어 정확히 사용. 정중하고 겸손한 톤. 구체적 안내 포함.

[정책 - DB에서 동적 조회]
- 영업시간: 평일 10:00~13:00, 14:30~17:00
- 배송 기간: 주문 확정 후 2-3일 내 발송. 국제배송 10-12일
- 배송 추적: 발송 완료 후 추적번호 메일 안내

[대화 이력 - 최근 메시지]
고객: 配送はいつ届きますか？ (배송 언제 오나요?)
```

시스템 프롬프트는 코드에 고정, 정책은 DB에서 조회, 대화 이력은 CsMessage에서 조회. 세 가지를 조합해서 Claude에 전달한다.

### LLM 채팅으로 답변 수정

AI가 생성한 답변이 마음에 안 들면 수정을 요청한다.

```
AI: "배송 조회 도와드릴게요"
상담원: "좀 더 정중하게 해줘"
AI: "배송 상황을 확인하여 안내 도와드리겠습니다"
상담원: "사과 표현도 추가해줘"
AI: "기다려주셔서 감사합니다. 배송 상황을 확인하여..."
```

최종적으로 "번역하여 보내기"를 누르면 ko→jp 번역 후 LINE으로 전송된다.

## 아키텍처 전체 구조

```
LINE 서버
    │ Webhook
    ▼
API Gateway → ALB → EC2 (NestJS)
                      │
                      ├── LINE 모듈: 웹훅 수신, 서명 검증, 메시지 전송
                      ├── Claude 모듈: 번역, 답변 추천, LLM 채팅
                      ├── Ticket 모듈: 상태 관리, CRUD
                      └── Slack 모듈: 알림 발송
                      │
                      ▼
                    MySQL
              (CsUser, CsTicket, CsMessage,
               CsCategory, CsPolicy)
```
