# 자동 급식기, 왜 같은 시간에 먹지 못할까?

집에 고양이가 두 마리 있다.<br>
각각 다른 방에서 생활하고 있어서 자동 급식기도 두 대를 쓰고 있다.<br>

같은 시간으로 설정해뒀는데, 어느 날 문득 이상한 걸 느꼈다.<br>

급식기 A가 먼저 돌아가고, 한참 뒤에 B가 돌아간다.<br>

시간으로 따지면 대략 이런 차이다.<br>

- 급식기 A → 12:00:01
- 급식기 B → 12:00:31

30초.<br>

사람 입장에서는 별것 아닐 수 있는데,<br>
고양이 입장에서는 꽤 큰 차이다.<br>

먼저 밥 소리가 나면 다른 쪽 고양이가 반응하고,<br>
자기 밥이 아직 안 나왔으니 스트레스를 받는다.<br>

사소하지만, 매일 반복되면 사소하지 않다.<br>

## 사전에 시도해봤던 케이스

### 케이스 1. 메시지 큐 처리 시간 분산 시도

가장 먼저 한 건 급식기 업체에 문의하는 것이었다.<br>

![업체 CS 문의](/images/posts/iot-feeder-sync-1/cs-inquiry.png)

돌아온 답변은 명확한 원인 설명 없이 "정상 범위"라는 내용이었다.<br>

그래서 직접 원인을 추적하기로 했다.<br>

이 급식기 시스템은 메시지 큐 기반으로 동작하고 있었기 때문에,<br>
처음에는 단순히 큐에 작업(Job)이 몰리면서 처리 순서가 밀리는 문제라고 판단했다.<br>

정각이나 30분 단위는 아무래도 다른 사용자들의 예약이 집중될 가능성이 높으니,<br>
비교적 Job이 몰리지 않을 시간대로 분산시켜 봤다.<br>

- 12:17
- 12:21
- 12:42

하지만 결과는 동일했다.<br>

시간을 분산시켜도 여전히 기기 간 실행 시점에 차이가 발생했다.<br>

이걸 통해 하나는 확실해졌다.<br>

**메시지 큐 처리 자체가 병목은 아니다.**<br>

오히려 문제는 그 이후 구간에 있는 것으로 보인다.<br>

```
서버(메시지 큐) → [네트워크] → 기기 수신 → 기기 내부 처리 → 모터 동작
                  ↑ 여기서 기기마다 차이 발생
```

다만, 이 부분은 아직도 완전히 납득되지는 않는다.<br>

왜냐하면 동일한 시점에 동일한 요청을 보냈음에도 불구하고<br>
각 기기까지 도달하는 시간이 다르게 나타나기 때문이다.<br>

현재로서는 다음과 같은 요소들이 영향을 준다고 추정하고 있다.<br>

- 네트워크 경로 차이
- 기기별 Wi-Fi 상태
- 기기 내부 처리 지연

하지만 이 중 어떤 것이 결정적인 원인인지는 명확하지 않다.<br>

---

## 왜 이런 문제가 생길까?

대부분의 상용 급식기는 서버에서 명령을 보내는 구조로 동작한다.<br>

```
서버 → "지금 급식해" → 기기 A
서버 → "지금 급식해" → 기기 B
```

서버가 같은 시각에 명령을 보내더라도, 실제 실행 시점은 기기마다 다르다.<br>

실제로 메시지 큐 레벨에서의 지연을 의심해 분산 테스트를 진행했지만, 문제는 해결되지 않았다.<br>

원인은 단순하다.<br>

- **네트워크 지연**: Wi-Fi 상태에 따라 수십 ms에서 수 초까지 차이가 난다.
- **기기 응답 속도**: 슬립 모드에서 깨어나는 시간이 기기마다 다르다.
- **서버 처리 순서**: 명령이 순차적으로 나간다면, 뒤에 있는 기기는 그만큼 늦다.

결국 "동시에 보냈다"가 "동시에 실행된다"를 보장하지 못하는 구조다.<br>

---

## 명령 기반의 한계

이 문제의 본질은 하드웨어가 아니라 **시간 기준**에 있다.<br>

서버가 "지금 해"라고 말하는 방식은,<br>
네트워크 위에서 정확한 타이밍을 보장할 수 없다.<br>

급식기 입장에서는 명령이 도착한 시점이 곧 실행 시점이 되는데,<br>
그 "도착 시점"이 기기마다 다르니 당연히 어긋난다.<br>

---

## 목표

해결 방향은 간단하다.<br>

~~서버가 "지금 해"라고 명령을 보내는 방식~~ <br>
→ 서버가 **"12시에 해"라는 예약 정보를 내려주고, 기기가 스스로 실행하는 방식**<br>

**명령 기반 → 예약 실행 기반**으로 전환하는 것이다.<br>

각 기기가 동일한 시간 기준을 갖고 있다면,<br>
네트워크 상태와 무관하게 같은 시각에 실행할 수 있다.<br>

---

## 해결 전략

이걸 구현하려면 두 가지가 필요하다.<br>

1. **기기가 정확한 현재 시각을 알고 있어야 한다** → NTP 시간 동기화
2. **기기가 스스로 판단해서 동작을 실행할 수 있어야 한다** → ESP32 로컬 스케줄러

기존 급식기의 물리 버튼을 릴레이 모듈로 제어하면,<br>
급식기 자체를 뜯어 고치지 않고도 외부에서 버튼을 "눌러줄" 수 있다.<br>

---

## 전체 구조

```
앱 → 서버 → ESP32 → 릴레이 → 급식기 버튼
```

| 구간 | 역할 |
|------|------|
| 앱 | 급식 시간 설정 UI |
| 서버 | 스케줄 저장, ESP32에 예약 정보 전달 |
| ESP32 | NTP로 시간 동기화, 예약 시각에 릴레이 동작 |
| 릴레이 | 급식기 물리 버튼 전기 신호 제어 |
| 급식기 | 기존 상용 급식기 (개조 없음) |

핵심은 **서버가 실행 명령을 보내지 않는다**는 점이다.<br>
서버는 "몇 시에 급식할지"만 알려주고, 실제 실행은 ESP32가 한다.<br>

## 핵심 기술

### ESP32

Wi-Fi가 내장된 저전력 마이크로컨트롤러.<br>
Arduino 환경에서 개발할 수 있고, GPIO 핀을 통해 릴레이 같은 외부 모듈을 직접 제어할 수 있다.<br>

### NTP (Network Time Protocol)

인터넷을 통해 정확한 시각을 동기화하는 프로토콜이다.<br>
ESP32가 부팅되면 NTP 서버에서 현재 시각을 받아오고,<br>
이후에는 내부 클럭 기준으로 시간을 유지한다.<br>

기기마다 독립적으로 NTP 동기화를 하면,<br>
서로 통신하지 않아도 동일한 시간 기준을 갖게 된다.<br>

### 릴레이 모듈

전기 신호로 스위치를 제어하는 부품이다.<br>
ESP32의 GPIO 핀에서 HIGH 신호를 보내면 릴레이가 닫히면서 급식기 버튼이 눌린다.<br>

## 정리

이 문제는 급식기 성능의 문제가 아니라, **실행 시점을 누가 결정하느냐**의 문제였다.<br>

서버가 결정하면 네트워크 지연이 끼어들고,<br>
기기가 결정하면 네트워크 상태와 무관하게 실행할 수 있다.<br>

이 문제는 단순한 서버 처리 지연이 아니라,<br>
**서버에서 각 기기까지 전달되는 시간의 비결정성**에서 발생한다.<br>

**명령을 보내지 말고, 실행 시간을 내려라.**<br>

다음 글에서는 ESP32 펌웨어 구현과 NTP 동기화를 실제로 어떻게 구성했는지 다뤄보려 한다.<br>
