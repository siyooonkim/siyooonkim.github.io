일본 시장에서 LINE으로 CS를 운영하고 있었다. 문제는 단순했다.

- 고객은 일본어로 문의한다.
- CS팀은 한국어만 안다.
- CS 인력이 많지 않다.

구글 번역기를 켜고 복사-붙여넣기 하면서 응대하고 있었는데, 이건 아니었다. 번역 품질도 들쭉날쭉하고, 경어 틀리면 일본 고객은 바로 느낀다. 무엇보다 하나의 문의를 처리하는 데 시간이 너무 오래 걸렸다.

필요한 건 세 가지였다.

1. **실시간 번역** — 고객 메시지가 오면 자동으로 한국어로 보여주고, 상담원이 한국어로 쓰면 일본어로 변환해서 보내기
2. **AI 답변 추천** — 과거 응대 패턴과 브랜드 정책을 기반으로 답변 초안 생성
3. **데이터 축적** — 문의 유형별 분류, 태그, 응답 시간 등을 쌓아서 자동응답으로 확장할 기반

---

## 요구사항은 받는 게 아니라 만든다

처음 받은 요구사항은 이거 하나였다.

> LINE으로 들어오는 일본어 CS를 좀 효율적으로 처리할 수 있게 해주세요.

여기서 바로 구현에 들어가면 안 된다. "효율적으로"가 정확히 뭔지부터 정의해야 한다.

역질문을 던졌다.

- 상담원이 일본어를 아예 모르는 건가, 조금은 아는 건가?
- 문의 유형이 대체로 비슷한가, 다양한가?
- 고객이 보내는 건 텍스트만인가, 이미지도 있는가?
- 답변 톤은 어느 정도 격식을 갖춰야 하는가?
- 영업시간 외 문의는 어떻게 처리하고 있는가?

질문을 던질수록 실제 요구사항이 드러났다.

### 확정된 요구사항

| 항목 | 결정 |
|------|------|
| 채널 | LINE (일본 시장 메인 메신저) |
| 번역 | 실시간 양방향 (jp↔ko), AI 번역 (Claude) |
| 상담 방식 | 봇이 카테고리 분류 → 상담원 연결 |
| AI 어시스트 | 대화 이력 + 정책 기반 답변 추천, LLM 채팅으로 수정 가능 |
| 통계 | 카테고리별, 태그별, 담당자별 분류 → 자동응답 확장 기반 |
| 알림 | 새 문의 시 Slack 알림 |

## 유저 플로우

시스템이 어떻게 돌아가는지, 두 가지 관점에서 그렸다.

### 고객 관점: LINE에서 문의하기

```
일본 고객 (LINE)                         시스템
    |                                      |
    |--- "配送はいつ届きますか？" --------->|
    |                                      |-- 유저 확인 (신규면 프로필 조회 후 생성)
    |                                      |-- 티켓 생성
    |                                      |-- 메시지 저장 + jp→ko 번역
    |                                      |
    |<-- 인사말 + 카테고리 선택 버튼 ------|
    |    "お問い合わせありがとうございます    |
    |     カテゴリをお選びください"          |
    |                                      |
    |--- "📦 配送" 버튼 탭 -------------->|-- 카테고리 설정
    |--- "購買後" 서브 탭 --------------->|-- 서브카테고리 설정
    |                                      |
    |<-- "担当スタッフにおつなぎします"  ---|-- Slack 알림 발송
    |                                      |
```

고객이 메시지를 보내면 봇이 카테고리를 선택하게 하고, 선택이 끝나면 상담원에게 넘긴다. 상담원은 Slack 알림으로 새 문의를 인지한다.

### 상담원 관점: 어드민에서 응대하기

```
한국 상담원 (Admin)                      시스템
    |                                      |
    |--- 티켓 목록에서 대기 티켓 클릭 ---->|
    |<-- 채팅 타임라인 표시 --------------|
    |    (원문 jp + 번역 ko 동시 표시)     |
    |                                      |
    |--- [AI 답변 추천] 클릭 ------------>|-- Claude에 전달:
    |                                      |   대화 이력 + 정책 + 카테고리
    |                                      |
    |<-- AI 답변 제안 (ko + jp) ----------|
    |                                      |
    |--- "좀 더 정중하게 해줘" ----------->|-- LLM 채팅으로 수정
    |<-- 수정된 답변 --------------------|
    |                                      |
    |--- "번역하여 보내기" 클릭 ---------->|-- LINE Push 전송
    |                                      |
```

상담원은 한국어만 쓴다. AI가 번역하고, 답변도 추천한다. 상담원은 추천된 답변을 확인하고 필요하면 LLM 채팅으로 수정한 뒤 보낸다.

## 티켓 상태 머신

CS 시스템의 핵심은 티켓 상태 관리다. 상태가 엉키면 고객이 답변을 못 받거나, 같은 문의에 두 명이 달라붙는 사고가 난다.

```
┌──────────────┐
│ bot_handling  │ ← 최초 상태. 봇이 카테고리 안내 중
└──────┬───────┘
       │ 카테고리 선택 완료
       ▼
┌──────────────┐
│ waitingAgent │ ← 상담원 대기. Slack 알림 발송
└──────┬───────┘
       │ 상담원 첫 메시지 전송
       ▼
┌──────────────┐
│agentHandling │ ← 고객 답변 대기
└──────┬───────┘
       │ 상담 종료 (카테고리 + 태그 저장)
       ▼
┌──────────────┐
│   closed     │
└──────────────┘
```

| From → To | 트리거 | 누가 |
|-----------|--------|------|
| bot_handling → waitingAgent | 카테고리 선택 완료 | 시스템 |
| waitingAgent → agentHandling | 상담원 첫 메시지 | 상담원 |
| agentHandling → waitingAgent | 고객 추가 메시지 | 시스템 |
| waitingAgent/agentHandling → closed | 상담 종료 | 상담원 |

종료 시에는 **상담원 카테고리 + 태그**를 필수로 입력한다. 이 데이터가 나중에 자동응답 학습의 기반이 된다.

---

## 카테고리 설계: 왜 2단계인가

문의 유형을 분류하는 건 단순해 보이지만, 설계를 잘못하면 나중에 통계가 쓸모없어진다.

처음엔 카테고리 하나만 두려고 했다. "배송", "취소", "반품" 정도면 되지 않나? 하지만 실제 문의를 보면 같은 "배송"이어도 결이 다르다.

- "아직 안 왔어요" (구매 후 배송 추적)
- "얼마나 걸려요?" (구매 전 배송 기간 문의)
- "주소 변경하고 싶어요" (배송 기타)

이걸 전부 "배송"으로 묶으면 통계에서 아무것도 안 보인다. 그래서 **대분류 + 서브카테고리** 2단계 구조로 갔다.

| 대분류 | 서브카테고리 |
|--------|-------------|
| 상품 | 속성, 구성, 소재, 재입고, 무게 |
| 주문 | 기타, 해외 |
| 배송 | 구매전, 구매후, 기타 |
| 캔슬 | 기타, 불만, 접수, 품절, 환불 |
| 반품/교환 | 기타, 접수, 현황 |
| AS | AS |
| 기타 | 메일수신거부, 이벤트, 인스타, 쿠폰/포인트, 회원, 리뷰 |

고객이 LINE에서 카테고리를 선택하고, 상담 종료 시 상담원이 더 세밀한 분류를 붙인다. 두 관점의 데이터가 동시에 쌓이기 때문에, "고객이 생각하는 문의 유형"과 "실제 상담 내용"의 차이도 볼 수 있다.
