## DB 모델 설계

### 엔티티 관계

```
┌──────────┐  1:N  ┌──────────┐  1:N  ┌──────────┐
│  CsUser  │──────>│ CsTicket │──────>│CsMessage │
└──────────┘      └──────────┘      └──────────┘
                        │
              N:1       │ JSON
            ┌───────────┤
            │           │
      ┌──────────┐   tags[]
      │CsCategory│
      └──────────┘
```

고객(CsUser)이 문의하면 티켓(CsTicket)이 생기고, 메시지(CsMessage)가 쌓인다. 티켓에는 카테고리와 태그가 붙는다.

### 각 테이블의 역할

**CsUser** — LINE 유저 정보

```sql
lineUserId  VARCHAR(50)  -- LINE 발급 유저 ID
displayName VARCHAR(100) -- LINE 프로필 이름
language    VARCHAR(10)  -- 'jp'
```

LINE userId는 고유하다. 같은 고객이 다시 문의하면 기존 유저에 연결한다. displayName이 "LINE User"로 오는 경우가 있는데, 프로필 API를 한 번 더 호출해서 갱신한다.

**CsTicket** — 상담 티켓

```sql
userId      INT          -- FK -> CsUser
subject     TEXT         -- 첫 메시지 내용
status      ENUM         -- bot_handling, waitingAgent, agentHandling, closed
categoryId  INT          -- FK -> CsCategory (고객 선택)
agentCategoryId INT      -- 상담원이 종료 시 선택
tags        JSON         -- 상담원이 종료 시 입력 ["배송지연", "긴급"]
priority    ENUM         -- low, normal, high
assignedTo  VARCHAR(50)  -- 담당 상담원
resolvedAt  DATETIME     -- 종료 시각
```

핵심은 **카테고리가 두 개**라는 점이다. `categoryId`는 고객이 봇에서 선택한 카테고리, `agentCategoryId`는 상담원이 종료 시 직접 분류한 카테고리다.

고객이 "배송"을 선택했는데 실제 상담 내용이 "주소 변경 -> 취소 -> 재주문"이면, 상담원은 "주문변경"으로 분류한다. 이 차이가 쌓이면 "고객은 배송이라 생각하지만, 실제로는 주문 관련 문의가 많다"는 인사이트가 나온다.

**CsMessage** — 메시지

```sql
ticketId    INT          -- FK -> CsTicket
senderType  ENUM         -- 'customer' | 'agent' | 'bot'
content     TEXT         -- 원문
translated  TEXT NULL    -- 번역문 (null이면 번역 실패)
language    VARCHAR(10)  -- 원문 언어 ('jp' | 'ko')
```

메시지는 **append-only**다. 수정/삭제 없음. CS 대화 기록은 분쟁 시 증거가 될 수 있고, AI 학습 데이터로도 쓰인다. 원본이 변하면 안 된다.

`translated`가 null인 경우는 번역 실패다. 빈 문자열("")이 아니라 null로 구분하는 이유는, "번역이 안 된 건지, 번역할 내용이 없는 건지"를 구분하기 위해서다.

## 핵심 규칙

| 규칙 | 이유 |
|------|------|
| 고객당 활성 티켓 최대 1개 | 같은 고객의 문의가 여러 티켓으로 분산되는 것 방지 |
| 메시지 append-only | 증거 보전 + AI 학습 데이터 무결성 |
| 번역 실패 = null | 빈 문자열과 구분 |
| 종료된 티켓도 재오픈 가능 | 고객이 같은 건으로 추가 문의하는 경우 |
| 종료 시 카테고리 + 태그 필수 | 자동응답 학습 기반 데이터 |

---

## 데이터 축적 -> 자동응답 확장

이 시스템의 최종 목표는 "사람이 응대하되, 점점 자동화할 수 있는 기반을 쌓는 것"이다. 매 상담마다 두 가지 데이터가 쌓인다.

### 1. 카테고리 분류 데이터

고객 선택 카테고리와 상담원 분류 카테고리가 동시에 쌓인다.

```
상담 100건 축적 후:
- "배송 > 구매후" 문의 42건
- "캔슬 > 접수" 문의 23건
- "상품 > 재입고" 문의 15건
```

빈도가 높은 유형부터 자동응답 템플릿을 만든다. "배송 > 구매후" 문의의 80%가 "언제 오나요?"류라면, 봇 단계에서 추적번호와 예상 도착일을 바로 안내할 수 있다.

### 2. 태그 데이터

상담원이 종료 시 자유 입력하는 태그가 쌓인다.

```
태그 빈도:
- "배송지연" 38건
- "사이즈문의" 25건  
- "쿠폰오류" 18건
- "긴급" 12건
```

태그는 카테고리보다 세밀한 분류다. "배송지연"이 급증하면 물류 이슈를 감지할 수 있고, "쿠폰오류"가 반복되면 시스템 버그를 의심할 수 있다. CS 데이터가 운영 모니터링 역할도 한다.

### 확장 로드맵

```
[현재] 사람이 응대, 데이터 축적
   |
[Phase 1] 빈도 높은 유형에 답변 템플릿 추가
   |
[Phase 2] 봇 단계에서 FAQ 매칭 -> 자동 답변 시도
   |
[Phase 3] 자동 답변 실패 시에만 상담원 연결
```

지금은 Phase 0이다. 모든 문의가 상담원을 거친다. 카테고리 + 태그 + 대화 이력이 쌓일수록 자동화할 수 있는 영역이 명확해진다.

RAG를 처음부터 도입하지 않은 이유도 여기에 있다. 데이터가 충분히 쌓이기 전에 자동화를 서두르면 "자신 있게 틀린 답변"을 하는 봇이 된다. 사람이 먼저 응대하면서 데이터를 쌓고, 확실한 패턴이 보일 때 자동화하는 게 안전하다.

## 통계 대시보드

축적된 데이터를 보는 대시보드도 만들었다.

- **개요 카드**: 전체 문의, 종료 문의, 대기 문의, 평균 응답 시간
- **카테고리별 분포**: 고객 선택 vs 상담원 분류 각각
- **태그별 분포**: 빈도순 정렬
- **일별 추이**: 문의량 변화
- **담당자별 성과**: 처리 건수, 평균 응답 시간

평균 응답 시간은 `티켓 생성 시각`과 `상담원 첫 메시지 시각`의 차이로 계산한다. 이 수치가 올라가면 인력 충원이 필요하다는 신호다.

---

## 정리

전체 시스템을 한 줄로 요약하면:

> **LINE 웹훅으로 메시지를 받고, Claude로 번역하고, DB 정책을 프롬프트에 주입해서 답변을 추천하고, 카테고리와 태그를 쌓아서 자동응답의 기반을 만든다.**

RAG 같은 거창한 구조 없이도, 카테고리 기반 정책 주입 + 데이터 축적만으로 실용적인 CS 봇을 만들 수 있었다. 정책이 수백 개로 늘어나면 그때 RAG를 검토하면 된다. 지금은 이 구조로 충분하다.
