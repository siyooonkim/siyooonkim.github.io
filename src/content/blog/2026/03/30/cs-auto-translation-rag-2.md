[이전 편](/posts/2026/03/30/cs-auto-translation-rag-1)에서 요구사항을 정리하고 유저 플로우를 설계했다. 이번 편에서는 AI 부분을 설계한다.

핵심 질문은 하나였다. **AI가 추천 답변을 생성할때, 참고해야할 정책서와 응대 지식을 어떻게 주입할 것인가?**

---

## 두 가지 선택지

### 선택지 A: RAG (Retrieval-Augmented Generation)

RAG는 요즘 LLM 활용에서 자주 등장하는 패턴이다. 간단히 설명하면 이렇다.

```
1. 사내 문서(FAQ, 정책 가이드, 과거 응대 로그)를 벡터 DB에 저장
2. 고객 질문이 들어오면 → 벡터 검색으로 관련 문서를 찾음
3. 찾은 문서 + 질문을 LLM에 함께 전달
4. LLM이 문서 내용을 참고해서 답변 생성
```

장점은 분명하다. 문서가 많아져도 관련 있는 것만 골라서 프롬프트에 넣으니, 토큰 낭비가 적고 확장성이 좋다. 수백~수천 개의 정책 문서가 있는 대규모 CS센터라면 RAG가 맞다.

### 선택지 B: 과거 응대 참조 + DB 정책 주입

RAG를 쓰지 않고, 과거 유사 응대 이력과 카테고리별 정책을 DB에서 꺼내서 프롬프트에 직접 넣는 방식이다.

```
1. 고객 문의가 들어오면 → 같은 카테고리의 종료된 티켓에서 과거 유사 응대 사례를 우선 조회
2. 해당 카테고리의 정책을 JpCsPolicy 테이블에서 보조 조회
3. 과거 응대 예시 + 정책을 프롬프트에 직접 주입
4. LLM이 과거 사례를 우선 참고하고, 정책으로 보정하여 답변 생성
```

---

## RAG를 사용하지 않은 이유 

한 문장으로 말하자면 **우리 상황에서 RAG는 오버스펙이기 때문이다.**

### 정책 문서가 많지 않다

RAG가 빛나는 건 참조해야 할 문서가 수백~수천 건일 때다. 
벡터 검색으로 관련 문서를 골라내는 게 의미가 있으려면 그만큼 문서 풀이 커야 한다.

우리 정책은 100개가 안되기 때문에 벡터 검색 없이 카테고리 기준 DB 쿼리 한 번이면 필요한 정책을 전부 꺼낼 수 있다.

과거 유사 응대도 마찬가지다. 카테고리 분류가 잘 되어 있기 때문에, 같은 카테고리의 종료된 티켓을 SQL로 조회하면 충분히 유사한 사례가 나온다. 굳이 벡터화해서 의미적 유사도를 계산할 필요가 없다.

### 인프라 오버헤드

벡터 DB 구축, 임베딩 파이프라인 설계, 검색 품질 튜닝까지 들어가는 비용이 너무 크다고 판단했다.
(다른 프로젝트 목적으로 요새 백터 DB를 구축해서 파이프라인을 만들고 있는데, 예상했던 것보다 시간을 오래 잡아먹는중이다.)

### 정확도 문제

RAG는 "관련 있는 문서를 검색"하는 단계에서 100% 정확하지 않다. 
벡터 유사도 기반이라, 의미적으로 비슷하지만 실제로는 다른 정책을 가져올 수 있다.

다행히 **유저가 카테고리를 직접 선택**하도록 설계했기 때문에,
(배송을 선택하면 배송 정책을, 반품을 선택하면 반품 정책을)
별도 검색할 필요 없이, 유저가 선택한 카테고리 기반으로 정책서를 조회하면 그만이다. 



---

일본 CS에 특화된 정책을 별도 테이블로 분리했다. 카테고리별로 조회해서 프롬프트에 주입한다. 
정책의 추가, 수정, 삭제는 어드민 API를 만들어서 코드 배포 없이 직접 관리할 수 있는 구조로 만들었다.




### 과거 유사 응대 참조

정책만으로는 부족하다. 
브랜드 정책에서 확인이 필요한 경우에는 브랜드에 일일히 확인해야하는 비용이 또 들기 때문이다. 
그래서 브랜드 문의 최근 답변 중 같은 카테고리로 종료된 티켓의 응대 사례를 함께 넣는다면, 브랜드 소통 비용없이 CS팀에서 일괄적으로 처리할 수 있는 문의가 많아진다.

## AI 답변 추천 동작

### 프롬프트 구성

```
[시스템 프롬프트 - 고정]
당신은 일본 고객 CS 전문 상담 AI입니다.
일본어 경어 정확히 사용. 정중하고 겸손한 톤. 구체적 안내 포함.

[과거 유사 응대 예시 - CsTicket/CsMessage에서 카테고리 기반 조회]
Q: 注文した商品はいつ届きますか？
A: ご注文いただきありがとうございます。通常、ご注文確定後2〜3日以内に発送し、
   国際配送は10〜12日ほどお時間をいただいております。

[정책 테이블에서 카테고리 기반 조회]
- 영업시간: 평일 10:00~13:00, 14:30~17:00
- 배송 기간: 주문 확정 후 2-3일 내 발송. 국제배송 10-12일
- 배송 추적: 발송 완료 후 추적번호 메일 안내

[현재 대화 이력 - CsMessage에서 조회]
고객: 配送はいつ届きますか？ (배송 언제 오나요?)
```

과거 유사 응대가 프롬프트의 핵심이다. LLM은 이전에 같은 유형의 문의에 어떻게 답했는지를 먼저 참고하고, 정책은 사실 관계를 보정하는 용도로 쓴다. 이 순서가 중요한데, 과거 응대 예시가 톤과 답변 구조를 잡아주고, 정책이 정확한 수치와 조건을 채워준다.

### LLM 채팅으로 답변 수정

AI가 생성한 답변이 마음에 안 들면 자연어로 수정을 요청할 수 있다.

![AI 답변 수정 화면](/images/posts/cs-auto-translation-rag-2/llm-chat-ui.png)

"더 정중하게", "사과 표현 추가" 같은 빠른 수정 버튼도 만들어서 자주 쓰는 수정 요청은 클릭 한 번으로 처리할 수 있게 했다.

최종적으로 "번역하여 보내기"를 누르면 ko→jp 번역 후 LINE으로 전송된다.

## 통계 대시보드

축적된 데이터를 보는 대시보드도 만들었다.

- **개요 카드**: 전체 문의, 종료 문의, 대기 문의, 평균 응답 시간
- **카테고리별 분포**: 고객 선택 vs 상담원 분류 각각
- **태그별 분포**: 빈도순 정렬
- **일별 추이**: 문의량 변화
- **담당자별 성과**: 처리 건수, 평균 응답 시간

평균 응답 시간은 `티켓 생성 시각`과 `상담원 첫 메시지 시각`의 차이로 계산한다. 이 수치가 올라가면 인력 충원이 필요하다는 신호다.

---

## 정리

> **LINE 웹훅으로 메시지를 받고, Claude로 번역하고, 과거 유사 응대 + 정책을 프롬프트에 주입해서 답변을 추천하고, 카테고리와 태그를 쌓아서 추천 품질을 계속 높인다.**

RAG 같은 복잡한 구조 없이도, 카테고리 기반 정책 주입 + 과거 응대 참조만으로 실용적인 CS 봇을 만들 수 있었다. 
데이터가 쌓일수록 AI 추천 답변의 품질이 올라가는 구조를 만들어놨고, 현재보다 많은 정책서가 추가된다면 그 때 RAG를 사용하면 된다. 

기획부터 개발까지 대략 4~5일정도 걸린듯하다. 
능동적으로 문제를 먼저 파악해서 접근했고, 주변에서 실제로 이걸 사용하기 시작한 이후로 CS팀에서 응대 효율이 올라가고 이게 일본 매출 상승으로 이어지니 뿌듯했다.
