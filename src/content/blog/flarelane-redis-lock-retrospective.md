이커머스 서비스에서 "장바구니에 담은 상품이 할인됐어요!" 푸시 알림은 전환율을 끌어올리는 대표적인 CRM 장치다. 문제는 너무 단순하게 만들면 사용자 경험이 금방 무너진다는 점이다. 할인 이벤트가 발생할 때마다 푸시를 쏘면, 하루에 10개 상품이 할인되는 순간 푸시도 10개가 날아간다.

PM팀 요청은 짧았다.

> 하루에 1건, 가장 임팩트 있는 푸시만 보내주세요.

짧은 문장이었지만, 실제로는 요구사항보다 **정의되지 않은 판단 기준**이 훨씬 많았다. 이 글은 그 모호한 요청을 어떻게 스펙으로 바꿨고, 왜 Redis candidate record, Bull delayed job, 분산 락, 중복 감지까지 붙이게 됐는지를 기록한 글이다.

## 요구사항은 받는 게 아니라 만든다

처음 받은 요구사항은 사실상 아래 한 줄이었다.

> 장바구니에 담은 상품이 할인되면 푸시 보내주세요. 근데 너무 자주 보내면 안 되니까, 적당히 조절해주세요.

여기서 제일 먼저 한 일은 구현이 아니라 **역질문으로 모호함을 쪼개는 것**이었다.

- "적당히"가 정확히 몇 건인가?
- 여러 상품이 동시에 할인되면 어떤 걸 보내야 하나?
- 이미 발송 대기 중인데 더 큰 할인 후보가 생기면 갈아끼워야 하나?
- 할인 종료나 할인 축소가 발송 직전에 일어나면 어떻게 할 것인가?

질문을 던질수록 실제 요구사항이 드러났다.

### 최종적으로 굳힌 스펙

| 항목 | 결정 |
| --- | --- |
| 발송 주기 | 3일에 1건 |
| 우선순위 | 할인 변동폭(`gap`) 큰 순, 동률이면 판매량 순 |
| 발송 시점 판단 | 이벤트 발생 시점이 아니라 발송 시점 기준으로 재계산 |
| 원복 처리 | `current <= before`면 발송 안 함 |
| 대기 중 신규 후보 | 발송 시점에 전체 후보 중 top 1 재선정 |

핵심은 **이벤트 시점에 바로 확정하지 않는다**는 점이었다. Day 0에 A 상품을 보내기로 정해도, Day 2에 더 큰 할인 후보 B가 생기면 발송 시점에는 B가 선택돼야 했다.

## before, after, current를 왜 다 저장했나

처음엔 `after`만 있으면 될 줄 알았다. 하지만 발송 직전에 할인 상태를 다시 판단하려면 `before`가 필요했다.

예를 들어:

- 이벤트 발생 전 할인율: `before = 40`
- 이벤트 발생 시점 할인율: `after = 50`
- 실제 발송 시점 할인율: `current = 45`

이때 `current > before`니까 아직 보낼 수 있다. 반대로 `current <= before`면 사실상 원래 상태로 돌아온 것이므로 발송하면 안 된다.

그런데 여기서 끝이 아니었다. 나중에 동일 상품이 다시 더 큰 할인으로 내려가면 재발송 여부를 판단해야 했고, 그때는 `after`도 필요했다.

즉 이 구조가 됐다.

- `before`: 원래 상태와 비교하기 위한 기준점
- `after`: 직전 발송 혹은 이벤트 강도를 기억하기 위한 값
- `current`: 발송 직전 실제 상태

이 세 값을 나눠두니, 할인 원복/유지/추가 하락을 모두 판단할 수 있었다.

## 락 범위를 어디까지 잡아야 하나

가장 오래 고민한 건 **락 key 설계**였다. 문제는 "같은 상품 중복 처리"가 아니라, **같은 유저에게 중복 발송되는 상황**이었다.

![락 범위 비교](/images/posts/flarelane/lock-scope-comparison.svg)

처음 떠오른 선택지는 세 가지였다.

1. 상품 단위 락
2. 유저+상품 조합 락
3. 유저 단위 락

상품 단위 락과 유저+상품 조합 락은 둘 다 결국 실패했다. 상품 A와 상품 B가 동시에 할인되면 서로 다른 key를 잡기 때문에 같은 유저에게 발송이 겹칠 수 있기 때문이다.

요구사항은 "한 유저에게 3일에 1건"이었다. 그러면 보호해야 하는 대상도 유저다. 그래서 최종 key는 아래처럼 갔다.

```ts
flarelane-dispatch-lock:{userId}
```

이 설계의 의미는 명확했다. 한 유저에 대해서는 무조건 직렬 처리한다. 락 범위가 넓어져 병렬성이 조금 떨어지더라도, 여기서는 정합성이 성능보다 중요했다. 동시에 100개 상품이 할인되는 극단적 상황보다, 한 유저가 같은 날 CRM 푸시를 2번 받는 경험이 훨씬 나빴기 때문이다.

## 비관적 락을 선택한 이유

락 범위뿐 아니라 **락의 철학**도 결정해야 했다. 낙관적 락으로 갈지, 비관적 락으로 갈지.

| 판단 기준 | 우리 상황 | 결론 |
| --- | --- | --- |
| 충돌 시 롤백 가능? | 불가능. 푸시는 이미 유저 폰에 도달함 | 비관적 락 |
| 충돌 빈도 | 완전히 낮다고 보기 어려움 | 비관적 락 |
| 충돌 시 비용 | 중복 푸시 = UX 훼손 | 비관적 락 |
| 성능 요구 | 실시간 거래 시스템 수준은 아님 | 비관적 락 감수 가능 |

낙관적 락은 "일단 처리하고 나중에 충돌 감지" 전략이다. 그런데 푸시는 외부 시스템에 나간 뒤에는 되돌릴 수 없다. DB에 unique constraint를 걸어도 이미 발송은 끝난 뒤라 늦다.

이 케이스는 **충돌 후 복구**가 아니라 **충돌 자체를 막아야 하는 작업**이었다. 그래서 비관적 락으로 결정했다.

## 락 TTL은 왜 20초였나

처음에는 10초를 생각했다. DB 조회, Redis 접근, 외부 API 호출 정도면 넉넉하다고 봤다. 그런데 리뷰 피드백에서 바로 지적이 나왔다.

> 네트워크 지연이나 외부 API 지연이 10초를 넘기면 락이 만료되고, 다른 워커가 들어올 수 있습니다.

TTL이 너무 짧으면 중복 발송 위험이 커지고, 너무 길면 워커가 죽었을 때 복구가 느려진다. 결국 **정답이 있는 숫자라기보다 운영 가능한 타협값**이 필요했다.

그래서 이렇게 정리했다.

```ts
export const FLARELANE_DISPATCH_LOCK_TTL_MS = 20 * 1000;
```

정상 처리 시간은 대체로 2~5초 안쪽이고, 여유 마진을 더해 20초로 잡았다. 중요한 건 숫자 자체보다도, 이 값을 상수로 빼서 운영 중 조정 가능하게 해둔 점이었다.

장기적으로는 auto-extend가 더 좋은 패턴이다. 처리 중에는 TTL을 연장하고, 중간에 워커가 죽으면 연장이 멈춰 자동 만료되게 만드는 식이다. 이번 작업에서는 일정상 넣지 못했지만, 다음 개선 후보로 남겨뒀다.

## 락 경합과 인프라 장애는 다르게 다뤄야 한다

초기 코드에서 가장 위험했던 부분은 이거였다.

```ts
try {
  const lock = await this.redis.lock(key);
  // ...
} catch (error) {
  return null;
}
```

이렇게 해두면 **모든 예외가 락 획득 실패처럼 보인다**. 그런데 실제로는 두 종류가 섞여 있다.

- 정상적인 락 경합: 다른 워커가 이미 같은 유저 락을 잡고 있음
- 비정상적인 인프라 장애: Redis 연결 실패, 타임아웃, execution error

둘은 대응이 완전히 달라야 한다.

```ts
private isLockContentionError(error: unknown): boolean {
  return (
    error instanceof ResourceLockedError ||
    (error instanceof Error && error.name === 'ResourceLockedError')
  );
}
```

락 경합이면 스킵 후 다음 job을 예약하면 되지만, Redis 장애면 조용히 삼키면 안 된다. 그건 Bull이 재시도하고, 로그와 알림으로 드러나야 하는 문제다. 이 구분을 넣고 나서야 "정상 경쟁"과 "비정상 실패"를 다르게 볼 수 있게 됐다.

## 락을 못 잡았다고 끝내면 candidate가 방치된다

이 문제도 실제로 코드를 쓰다 보니 보였다.

시나리오는 단순하다.

- Worker A: 락 획득 성공 → 발송 완료
- Worker B: candidate는 저장했지만 락 획득 실패 → 그냥 종료

그러면 상품 B 후보는 Redis에 남아 있는데, 아무도 다시 주워가지 않는다. 즉 **발송 안 됐는데도 묻히는 candidate**가 생긴다.

그래서 락 획득에 실패한 유저에 대해서는 pending job이 없을 때 다음 delayed job을 예약하도록 했다.

```ts
await Promise.allSettled(
  lockFailedUserIds.map(async (userId) => {
    const hasPendingJob = await this.nextSendJobService.hasPendingJob(userId);
    if (!hasPendingJob) {
      await this.nextSendJobService.scheduleNext(userId);
    }
  }),
);
```

이 한 줄기 로직이 있어야 candidate가 방치되지 않는다. 락 경쟁은 정상 상황이지만, 그 때문에 데이터가 영원히 묻히는 건 정상 흐름이 아니기 때문이다.

## 락만으로는 완벽하지 않아서 2차 방어를 넣었다

분산 락을 걸었다고 해서 중복 발송 가능성이 0이 되는 건 아니다. TTL 만료 직전 워커가 느려지면 아래 같은 그림이 생긴다.

- Worker A: 락 획득 후 느리게 처리
- TTL 만료
- Worker B: 같은 유저 락 획득
- A와 B가 둘 다 푸시 발송

그래서 발송 기록을 저장할 때 **중복 발송 신호를 감지하고 Slack으로 알리는 2차 방어**를 넣었다.

```ts
async recordSent(userId, productId, before, after) {
  const key = `flarelane-sent:${userId}:${productId}`;
  const previousRecord = await this.redis.get(key);

  if (previousRecord && previousRecord.after >= after) {
    await this.slackService.notifyFlareLaneDuplicateSend({
      userId,
      productId,
      previousAfter: previousRecord.after,
      currentAfter: after,
    });
  }

  await this.redis.set(key, { before, after }, TTL);
}
```

완벽한 시스템을 기대하기보다, **이상 징후가 생겼을 때 빨리 눈치채는 구조**가 현실적이었다.

## 전체 흐름은 이렇게 정리됐다

![CRM 푸시 발송 흐름](/images/posts/flarelane/dispatch-flow.svg)

요약하면 아래 순서다.

1. 할인 이벤트 발생
2. 장바구니 유저별 candidate record 저장
3. 유저 단위 분산 락 획득 시도
4. 락 획득 + pending job 없음 → 즉시 발송
5. 락 획득 실패 → pending job 없으면 다음 job 예약
6. 발송 성공 후 sent record 저장 및 candidate 삭제
7. 다음 후보가 남아 있으면 3일 뒤 delayed job 예약

핵심은 모든 판단을 이벤트 시점에 끝내지 않고, **발송 시점 재계산 + 유저 단위 직렬화 + 다음 job 보장**의 조합으로 묶었다는 점이다.

## 느낀 점

이번 작업에서 제일 크게 느낀 건 세 가지였다.

### 1. 요구사항은 받는 게 아니라 만드는 것이다

PM이 준 건 한 문장이었다. 실제로 운영 가능한 스펙은 대부분 역질문으로 만들었다. 특히 할인 종료, 할인 축소, 더 강한 신규 후보 등장 같은 엣지케이스는 질문하지 않으면 절대 드러나지 않는다.

### 2. 락 범위는 “보호하려는 대상”에 맞춰야 한다

상품 단위 락이냐 유저 단위 락이냐는 기술 취향 문제가 아니었다. 우리가 보호하려는 것은 상품이 아니라 **유저당 발송 횟수 제한**이었다. 그래서 유저 단위가 맞았다.

### 3. 완벽한 차단보다 관측 가능성이 더 중요할 때가 있다

TTL 만료나 예외적인 지연까지 감안하면 중복 가능성을 100% 제거하는 건 어렵다. 그럴수록 중복 감지, Slack 알림, 재시도 분리 같은 **관측과 복구 장치**가 훨씬 중요해진다.

결국 이번 작업은 Redis 락을 잘 거는 방법보다, **모호한 요구사항을 발송 가능한 스펙으로 만들고, 롤백 불가능한 외부 작업을 어떻게 안전하게 둘러쌀 것인가**에 더 가까운 작업이었다.
