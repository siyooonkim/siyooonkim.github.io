# '버스벨' 개발 (4) _ 앱 개발

백엔드를 만들었으니 이제 앱이다.

앱 개발은 생각보다 빠르게 진행됐다. 화면도 많지 않고, 거의 와이어프레임 수준을 그대로 넣었기 때문에. 디자인 패턴은 재사용성을 고려해서 Atomic 패턴으로 구성했다.

## React Native + React Query

처음에는 Swift로 iOS 네이티브 앱을 만들까 잠깐 고민했다. 근데 나중에 Android도 지원하고 싶은 욕심이 있었고, 무엇보다 앞에서도 언급했듯이 개발 생산성을 극대화시킬 수 있을 것 같아서 RN을 선택했다.

또한 React Query를 함께 사용했다. RN과 호환도 잘될 뿐더러 서버 상태 캐싱, 로딩/에러 처리, 리페칭이 전부 알아서 된다. 화면 코드가 깔끔해지는 게 체감이 확실히 된다.

## NativeWind를 걷어낸 이유

처음에는 NativeWind(React Native용 Tailwind CSS)를 세팅했다. Next.js에서 Tailwind 쓸 때 생산성이 좋았어서 당연히 RN에서도 좋을 줄 알았다.

근데 막상 써보니 불편한 점이 있었다. RN의 StyleSheet과 충돌하는 부분이 있고, 디버깅할 때 스타일이 어디서 적용되는지 추적이 어려웠다. 특히 RN은 웹과 달리 Flex 기본 방향이 column이라 Tailwind의 유틸리티 클래스가 직관적이지 않은 경우가 많았다.

결국 NativeWind를 통째로 걷어내고 React Native 기본 StyleSheet으로 돌아왔다. 화면이 4개밖에 안 되는 앱에서 Tailwind은 오버스펙이었다. 설정 복잡도만 늘리고 실질적인 생산성 향상은 크지 않았다.

## 화면 구성

PRD에서 화면 3~4개면 된다고 했는데, 진짜로 4개다.

```
Onboarding → Search → RouteDetail → AlarmList
```

| 화면 | 하는 일 |
|------|--------|
| Search | 버스 번호 입력 → 노선 선택 |
| RouteDetail | 정류장 목록 보고 → 알림 설정 |
| AlarmList | 내 알림 확인/취소 |
| Onboarding | 첫 진입 시 서비스 소개 |

사용자 입장에서 검색 → 선택 → 알림 설정, 3스텝이면 끝이다. 예전에 소개팅 앱을 만들다가 지쳐서 포기했던 기억이 있다. 처음부터 기능을 욕심내면 안 된다는 걸 배웠기 때문에, P0을 우선적으로 구현하는 데 집중했다.

## API 레이어

API 호출 코드를 화면 컴포넌트에 직접 때려넣는 건 피하고 싶었다. 나중에 엔드포인트 하나 바뀌면 화면 코드를 다 뒤져야 하니까.

`src/api/` 폴더에 도메인별로 나눴다.

```
src/api/
├── client.ts           # Axios 인스턴스, baseURL, 인터셉터
├── authApi.ts          # 로그인, 회원가입
├── busApi.ts           # 버스 검색, 노선 조회, ETA
├── notificationApi.ts  # 알림 생성, 조회, 취소
└── userApi.ts          # 프로필
```

화면에서는 React Query 훅으로 감싸서 쓴다. API 레이어 → React Query 훅 → 화면 컴포넌트. 이 구조가 깔끔하고 유지보수하기 편하다.

## 로그인 — 모달에서 별도 플로우로

처음에는 로그인/회원가입을 모달로 만들었다. 검색 화면 위에 모달이 뜨는 방식.

직접 써보니까 어색했다. 모달이 뜨면 뒤에 검색 화면이 반투명하게 보이는데, 유저 입장에서 "지금 내가 뭘 해야 하는 거지?" 하는 느낌이었다. 회원가입 폼이 모달 안에 들어가니 공간도 좁고.

결국 온보딩 → 로그인/회원가입 → 메인 순서로 별도 화면 플로우를 만들었다. 인증이 끝나야 메인 화면으로 넘어가는 구조. 이게 흐름이 훨씬 자연스럽다.

## 푸시 알림

앱에서 가장 삽질을 많이 한 부분이다.

3편에서 만든 백엔드의 알림 타이머가 조건을 충족하면, Firebase Cloud Messaging(FCM)을 통해 푸시를 쏜다. 흐름 자체는 단순하다.

1. 앱 시작 → Firebase에서 FCM 토큰 발급
2. 로그인할 때 이 토큰을 백엔드 `User` 테이블에 저장
3. 백엔드에서 ETA 조건 충족 → FCM으로 푸시 발송
4. 앱에서 수신

실제로 푸시가 오면 이렇게 뜬다.

```
390번 버스 곧 도착
벽산아파트에 약 2분 후 도착 예정입니다.
```

이거 뜨는 순간이 제일 뿌듯했다. 내가 만든 백엔드가 공공 API를 폴링하다가, 조건이 맞으면 Firebase를 거쳐서 내 폰에 알림을 쏴주는 거다. 한 사이클이 돌아가는 게 눈에 보이니까.

근데 문제는 앱 상태에 따라 푸시 처리 방식이 전부 다르다는 거다.

| 상태 | 처리 방식 |
|------|----------|
| 포그라운드 | 앱이 열려있을 때. 직접 핸들링 필요 |
| 백그라운드 | 앱이 뒤로 가있을 때. 시스템이 알아서 보여줌 |
| 종료 상태 | 앱이 꺼져있을 때. 시스템 트레이에 표시 |

특히 iOS가 까다로웠다. data-only 메시지는 백그라운드에서 제대로 안 받는 경우가 있어서, notification 필드와 data 필드를 같이 보내는 방식으로 우회했다.

이 부분은 공식 문서만 봐서는 알기 어렵고, 실제로 디바이스에서 테스트해보면서 하나씩 잡아야 한다.

## 마치며

버스벨은 거창한 서비스가 아니다. 그냥 아침에 더위 먹으면서 버스 기다리기 싫어서 만든 앱이다.

근데 만들다 보니 공공 API 연동, 실시간 폴링, FCM 푸시 파이프라인, React Native 빌드, 앱스토어 출시까지 한 사이클을 전부 경험하게 됐다.

완벽하진 않다. Redis로 타이머를 관리하면 더 견고하겠고, 지도 위에 버스 위치를 실시간으로 찍어주면 더 좋겠고, 지하철도 지원하면 더 좋겠다.

근데 중요한 건 일단 출시했다는 거다. 완벽해질 때까지 기다리면 영원히 못 낸다. 그건 여러 번 겪어봐서 안다.
