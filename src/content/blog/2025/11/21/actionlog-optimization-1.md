타 팀에서 요청하는 쿼리를 추출해줘야하는 경우가 종종 있는데,<br>
개발만 하다가 직접 쿼리문을 치면서 데이터를 뽑는게 가끔 재미(?)있을 때가 있다.<br>

(현재 작업에서 매우 고통받고 있다는 뜻이다.)<br>

이번에는 유저 ActionLog 기반의 테이블 기반에서 데이터를 추출해야했다.<br>

쿼리를 돌려놓은지 1시간이 지났는데도 추출되지 않았고, 뭔가 근본적인 문제가 있다는 신호로 느껴졌다.<br>

해당 ActionLog 테이블을 보니, 이미 수십억 건이 쌓여있었고, 기간 조회가 가장 빈번한 패턴임에도 날짜 기반 인덱스가 걸려있지 않았다.<br>

CRM 스쿼드 팀원들과 논의해보았을 때, 유저의 ActionLog 데이터가 에서 활용도가 굉장히 높다는걸 알게 되었고<br>
ActionLog 테이블을 **인덱스 최적화 → 쿼리 수정 → 파티셔닝** 3단계에 걸쳐 최적화 하는 작업을 진행해보았다.<br> 

---

## 전체 최적화 로드맵


1. **현황 분석** — 측정 기준 쿼리 선정, 인덱스 전수 조사
2. **인덱스 최적화** — B-Tree 인덱스 추가, 복합 인덱스 설계
3. **쿼리 정비** — 쿼리 전수조사, DATE() 함수 제거
4. **파티셔닝** — 월별 파티션, dual-write 전환

**가장 비용이 낮고 효과가 큰 것부터** 적용하자는 전략으로 순서를 정했다.<br>
인덱스 추가는 서비스 중단 없이 할 수 있지만, 파티셔닝은 테이블 구조 자체를 바꿔야 한다.<br> 

---

## ActionLog가 뭔데?

ActionLog는 말 그대로 **유저의 행동을 기록하는 테이블**이다.<br>
유저가 무엇을 클릭하는지에 따라 해당 이벤트가 로그로 쌓이는 거다.<br>
이걸 기반으로 유저 집단을 구성할 수 있고, 해당 집단을 대상으로 마케팅 플레이를 하여 주문 발생을 늘리는게 일반적으로<br>
마케팅 팀에서 다루는 작업중 하나다.<br>

문제는 서비스를 10년 운영하다 보니 이게 수십억 건이 쌓였다는 것이다.<br>
이 정도 규모가 되면 아무리 단순한 `SELECT`도 가볍지 않다.<br>

---

## 왜 1시간이나 걸렸을까?

아래 "최근 6개월간 A 행동을 5번 이상 했거나, B 행동을 3번 이상 한 유저 목록"을 뽑는 쿼리다.<br> 

```sql
SELECT userId
FROM ActionLog
WHERE type IN (A, B)
  AND createDate >= DATE_SUB(NOW(), INTERVAL 6 MONTH)
GROUP BY userId
HAVING SUM(CASE WHEN type = A THEN 1 ELSE 0 END) >= 5
    OR SUM(CASE WHEN type = B THEN 1 ELSE 0 END) >= 3;
```

### 근본 원인: createDate 인덱스가 없었다

얼핏 보면 평범한 쿼리지만, **기간 조회의 핵심인 `createDate`에 인덱스가 없으면** 풀 테이블 스캔을 하게 된다.<br> 

![과부하](/images/posts/actionlog-optimization-1/overload.png)

도서관에 비유하면, 1층부터 꼭대기 층까지 모든 책장을 하나하나 훑으면서 "이 책 맞나? 아닌데. 이건? 이것도 아닌데."를 반복하고 있던 셈이다.<br>근데 그 꼭대기가 눈에 보이지 않을 정도로 높은 상황이랄까.<br>

곧바로 ActionLog 테이블에 `createDate` 인덱스를 추가했다.<br>인덱스 하나만으로도 성능을 극적으로 개선할 수 있다.<br>

다만 인덱스를 남용하면 안 된다.<br>인덱스는 읽기를 빠르게 해주는 대신 **쓰기(INSERT/UPDATE) 시 인덱스도 함께 갱신해야 하기 때문에 쓰기 성능이 떨어진다.**<br>ActionLog처럼 쓰기가 빈번한 테이블이라면 특히 주의해야 한다.<br>다만 ActionLog의 INSERT는 단건 append 위주라 인덱스 갱신 비용이 크지 않다고 판단했고, 읽기 성능 개선 효과가 압도적이라 추가하기로 했다.<br>

---

## B-Tree: 인덱스가 빠른 진짜 이유

여기서 잠깐.<br>"인덱스를 추가하면 빨라진다"는 건 많이 들어봤을 것이다.<br>그런데 **왜** 빨라지는 걸까?<br>이걸 이해하려면 인덱스의 실체인 **B-Tree**를 알아야 한다.<br>

### 전화번호부로 이해하기

전화번호부를 생각해보자.<br>30억 명이 수록된 전화번호부에서 "김시윤"을 찾는다고 해보자.<br>

**방법 1: 처음부터 한 명씩 읽기 (= Full Table Scan)**

1페이지부터 시작해서 "강OO? 아니야. 고OO? 아니야. 구OO? 아니야..." 를 30억 번 반복한다. 이게 지금 우리 쿼리가 하고 있던 일이다.

**방법 2: 색인을 타고 가기 (= Index Scan)**

"ㄱ" 섹션으로 간다. → "김" 섹션으로 간다. → "김시" 구간으로 간다. → "김시윤" 찾았다. 3~4번이면 끝이다.

### B-Tree 구조

데이터베이스의 인덱스가 바로 이 색인 역할을 하는데, 그 내부 구조가 **B-Tree(Balanced Tree)**다.<br>


- **균형 트리**다. 어떤 데이터를 찾든 트리의 깊이(=탐색 횟수)가 동일하다.
- 10억 건이어도 트리 깊이는 대략 **4~5단계**에 불과하다.

즉, 디스크를 4~5번만 읽으면 원하는 데이터의 위치를 알 수 있다.<br>

| 데이터 건수 | Full Scan (O(n)) | B-Tree (O(log n)) |
|-----------|-----------------|-------------------|
| 1,000 | 1,000번 | ~10번 |
| 1,000,000 | 1,000,000번 | ~20번 |
| 1,000,000,000 | 1,000,000,000번 | ~30번 |

이 차이가 **1시간 vs 수십 초**의 차이를 만든다.<br>
---

### 무엇을 추가할까 

무작정 `createDate`만 추가하는 게 아니라, `type + createDate` 복합 인덱스도 함께 추가했다.<br>

ActionLog 관련으로 요청받은 쿼리들을 분류해보면, 대부분이 `WHERE type = ? AND createDate >= ?` 형태였다.<br>type과 createDate를 동시에 거르는 패턴이 압도적이었기 때문에, 두 컬럼을 묶은 복합 인덱스가 효과적이라고 판단했다.<br>

```sql
-- 1순위: createDate 단독 인덱스
ALTER TABLE ActionLog
ADD INDEX idx_createDate (createDate),
ALGORITHM=INPLACE, LOCK=NONE;

-- 2순위: type + createDate 복합 인덱스
ALTER TABLE ActionLog
ADD INDEX idx_type_createDate (type, createDate),
ALGORITHM=INPLACE, LOCK=NONE;
```

`createDate` 단독 인덱스만으로도 기간 조회는 빨라진다.<br>하지만 우리 쿼리 대부분이 `WHERE type = ? AND createDate >= ?` 형태였다.<br>이 경우 복합 인덱스가 있으면 MySQL이 **인덱스 하나로 두 조건을 동시에 걸러낼 수 있다.**<br>

```
idx_createDate만 있을 때:
  1. createDate로 6개월치 필터링     ← 인덱스 사용 ✅
  2. 그 중에서 type 필터링            ← 추가 스캔 필요 ⚠️

idx_type_createDate가 있을 때:
  1. type = A이면서 6개월 이내        ← 인덱스 하나로 끝 ✅✅
```

복합 인덱스에서 컬럼 순서가 `(type, createDate)`인 이유도 있다.<br>B-Tree 안에서 데이터가 **왼쪽 컬럼 기준으로 먼저 정렬**되고, 같은 값 내에서 오른쪽 컬럼으로 정렬된다.<br>

```
B-Tree 내부 (type, createDate) 정렬 순서:

type=A, 2024-01-01
type=A, 2024-01-02
type=A, 2024-01-03
...
type=B, 2024-01-01   ← type=B 구간 시작
type=B, 2024-01-02
type=B, 2025-10-21   ← 여기서부터 6개월 이내
type=B, 2025-10-22
...
type=C, 2024-01-01   ← type=C 구간 시작
type=C, 2025-10-21   ← 여기서부터 6개월 이내
type=C, 2025-10-22
...
```

`WHERE type IN (A, B) AND createDate >= 6개월 전` 쿼리는 이 정렬된 B-Tree에서 **type=A 구간의 최근 6개월 + type=B 구간의 최근 6개월** 딱 두 범위만 읽으면 된다.<br>수십억 건 전체를 훑는 것과는 차원이 다르다.<br>

---

그렇다고 인덱스를 무한정 추가하면 안 된다.<br>

인덱스가 많아질수록 INSERT/UPDATE마다 모든 인덱스를 갱신해야 하기 때문에 쓰기 성능이 떨어진다.<br>
**테이블이 읽기 위주인지 쓰기 위주인지를 먼저 판단**해야 하고, 쓰기가 많더라도 ActionLog처럼 단건 append 위주라면 **인덱스 갱신 비용**이 크지 않아서 충분히 감수할 만하다.<br>


## 인덱싱 후 결과

인덱스를 추가하고, 같은 쿼리를 돌려봤다.<br>

```
console_69  51 s
```

**1시간+ → 51초.<br>약 70배 개선.**<br>

인덱스 두 개 추가한 것뿐인데, 70배 차이가 난다.<br>B-Tree의 힘이다.<br>

인덱스 덕분에 30억 건을 풀스캔하지는 않지만, 6개월치 데이터 자체가 수천만 건이라 여전히 읽어야 할 양이 많다.<br>

여기서 더 줄이기 위해 다음 편에서는 파티셔닝 전 작업을 진행한다.<br>
